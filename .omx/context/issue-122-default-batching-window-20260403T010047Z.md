# Context Snapshot — issue #122 default batching window

- task statement: Plan clawhip issue #122 to add a short default batching window for bursty notifications.
- desired outcome: A minimal, mergeable first slice that groups routine burst notifications by default, keeps the window configurable, and avoids delaying urgent/failure paths.
- stated solution: Add configurable default batching around 5 seconds without building a large config system.
- probable intent hypothesis: Reduce Discord gateway/message churn and make routine bursts less noisy, especially around mentions, while preserving timely delivery for failures.
- known facts/evidence:
  - The dispatcher already centralizes event intake, ticker-driven flushes, and outbound sink delivery in `src/dispatch.rs:17-155`.
  - clawhip already has one specialized batcher for GitHub CI runs in `src/dispatch.rs:165-409`, proving dispatcher-level batching is an established pattern.
  - The daemon wires the dispatcher from config-derived dispatch settings in `src/daemon.rs:59-62`.
  - Dispatch config currently exposes only `ci_batch_window_secs`, with defaulting and validation in `src/config.rs:86-103` and `src/config.rs:573-576`.
  - Mentions are currently attached at route-resolution/render time and prepended into the final content in `src/router.rs:104-145`, so burst batching must account for mention behavior explicitly.
  - Existing source-side grouping already exists for tmux keyword windows (`src/source/tmux.rs:603-622`, `src/source/tmux.rs:641-697`) and workspace debounce (`src/source/workspace.rs:228-233`, `src/source/workspace.rs:245-299`), but those are source-specific rather than delivery-wide.
  - Git commit pushes can already collapse multiple commits into one event before dispatch in `src/events.rs:333-365` and render as grouped output in `src/render/default.rs:616-666`.
- constraints:
  - Keep scope tight and mergeable.
  - Config changes must remain backward-compatible.
  - No giant config system for the first slice.
  - English in code/comments/PR.
  - Tests should cover grouped delivery behavior.
  - Window must be configurable, not hard-coded; default should remain around 5 seconds.
- unknowns/open questions:
  - Which deliveries count as "routine" for v1: likely non-alert, non-failure Discord sends, but exact bypass set needs review.
  - Whether the first slice should batch only Discord sends or all sinks.
  - Whether mentions on grouped routine batches should be fully suppressed or conditionally retained when only one message remains in the batch.
- decision-boundary unknowns:
  - OMX may choose the narrowest dispatcher-level insertion point and a minimal dispatch config field name.
  - OMX should not expand into per-route batching policy unless the current code makes a single dispatch-level knob unsafe.
